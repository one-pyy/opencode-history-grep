# opencode-history-grep — 视图编译设计

状态：**未实现**

## 设计目标

编译阶段负责把 upstream 历史里的 message/part 结构，整理成更适合搜索和阅读的 block 视图。

这里的“编译”不是为了生成可执行物，而是为了把原始历史转换成：

- 更稳定的检索粒度
- 更清晰的角色边界
- 更容易生成摘要的中间表示

## 输入模型

状态：**已调研输入来源，未实现编译器**

上游历史已经区分为：

- session
- message
- part

其中 part 可能包含：

- `text`
- `tool`
- `file`
- `compaction`
- `subtask`
- `system`
- 其他运行时相关 part

编译器不直接暴露底层 part，而是先把这些 part 合并成 block。

## 输入过滤边界

并非所有 upstream part 都会进入本项目的编译仓库。

明确排除规则：

- `system` 不进入 compiled repository
- 因此也不会出现在 full view、search view、anchor 或 grep 结果中

这里的设计意图不是“结果页默认不显示 system”，而是把它排除在编译边界之外。

## Block 作为最小检索单位

第一版采用 **block** 作为搜索与展示的最小单位，而不是 message，也不是单行文本。

原因：

- message 太粗，不利于精准命中
- 单行文本太碎，无法保留上下文
- block 既能保留角色语义，也能保留局部上下文

## 建议的 block 类型

第一版建议至少支持以下 block 类型：

1. `user_text`
   - 用户文本输入块

2. `assistant_text`
   - 助手可见回答块

3. `tool_call`
   - 工具调用块，保留工具名和关键输入

4. `tool_result`
   - 工具结果块，保留结果摘要和必要错误信息

5. `compaction`
   - 历史压缩相关块

6. `subtask`
   - 子任务/子 agent 调用块

第一版不要求 block 类型绝对完整，但必须保证：

**用户看到搜索结果时，能一眼区分“谁说的”“这是什么行为”“结果是什么”。**

`system` 不属于第一版 block 类型集合，因为它不会进入编译仓库。

## 建议的 block 结构

每个编译后的 block 建议至少包含：

- `session_id`
- `message_id`
- `block_id`
- `role`
- `block_type`
- `title`
- `summary`
- `search_text`
- `display_text`
- `anchors`

### 字段说明

#### `summary`

给人快速判断的短概要。它是搜索结果质量的核心字段。

#### `search_text`

用于搜索匹配的归一化文本。可以比 `display_text` 更完整，但不直接暴露给用户。

#### `display_text`

用户在结果页实际看到的正文片段。

#### `anchors`

用于回跳到完整视图的定位信息。第一版不要求绑定 UI，只要求有稳定定位依据。

## 编译原则

### 1. 角色优先

先保留 user / assistant / tool 这些角色边界，再考虑文本截断与摘要。

### 2. 工具交互成对可读

工具调用和工具结果即使不强制合并，也必须在完整视图中能明显关联；搜索结果至少要让人知道“这是调用”还是“这是结果”。

### 3. 工具块默认保留调用参数

`tool_call` 默认保留调用参数，除非参数本身长到明显破坏可读性。原因是调用参数通常比输出更适合检索与判断语义。

### 4. 工具输出默认截断，但命中附近保留局部上下文

`tool_result` 不默认保留整段输出。

默认规则：

- 如果输出**不包含** grep 命中：只保留摘要与必要错误信息
- 如果输出**包含** grep 命中：默认保留命中附近的局部窗口

局部窗口规则：

- 优先按“命中上下各 5 行”取窗口
- 同时计算“命中前后各 500 字符”的窗口
- 实际保留范围取两者中**更小**的那个

这个规则的目标不是尽量多保留，而是避免工具结果一旦命中就把整段大输出灌回结果页。

### 5. 摘要优先于全文转储

默认展示不应当把超长工具输入或超长工具输出整段吐给用户。默认结果应该先给摘要，必要时再附有限片段。

### 6. 保留可回跳性

即使做降噪和截断，也不能失去跳回完整上下文的能力。

### 7. 编译仓库只保留检索主体

compiled repository 不是 upstream 全量镜像。

第一版只保留会参与历史检索与阅读的主体 block；`system` 这一类运行时噪音不进入仓库。

## 两类编译产物

### Compiled Full View

完整块视图，按时间顺序排列，适合阅读与回跳。

### Compiled Search View

命中块视图，适合回答“哪些历史片段与查询最相关”。

Search View 不应该重新从原始历史现拼，而应建立在 Full View 的 block 产物之上。

## 第一版刻意不做的事

- 不做 embedding 或向量检索
- 不做自动记忆注入
- 不做复杂跨 session 语义聚类
- 不做长期缓存设计优化

第一版只追求：

**把 upstream 历史稳定编译成高质量 block，并在 block 上做高信号搜索。**
