# opencode-history-grep — 命令设计

状态：**已实现主行为，`and/or` query 逻辑待落地**

## 当前默认入口

日常使用默认直接运行全局命令：

```bash
opencode-history-grep grep --query "keyword"
```

当前默认路径：

- upstream SQLite：`/root/.local/share/opencode/opencode.db`
- compiled repository：`/root/.local/share/opencode-history-grep`

`grep` 和 `show` 会自动 compile / refresh，一般无需手动先运行 `compile`。

自动 refresh 当前使用 repository 级 `.compile.lock` 做跨进程互斥；同一个 compiled repository 同一时间只允许一个进程执行编译，避免多个 `show` / `grep` 同时刷新时互相覆盖 `manifest.json`。

## 全局安装维护规则

本项目的 CLI 面向全局命令 `opencode-history-grep`。

因此每次修改 CLI 行为、参数、默认路径或输出格式后，都应刷新全局安装，使全局命令跟上本地源码。

推荐命令：

```bash
uv tool install --editable "/root/_/opencode/my_plugins/opencode-history-grep"
```

如果已经装过，也可先卸载再重装：

```bash
uv tool uninstall opencode-history-grep
uv tool install --editable "/root/_/opencode/my_plugins/opencode-history-grep"
```

## 工具形态

这是一个 Python CLI 工具，不是服务端进程，不是插件 hook，也不是 Web 界面。

第一版目标是让它能在本地直接运行，并面向 opencode upstream 历史做“编译 + 搜索”。

## 命令设计原则

命令层必须服务两个核心动作：

1. 编译历史
2. 搜索历史

第一版不追求命令很多，只追求清楚。

## 建议命令

### `compile`

作用：

- 读取指定 session 或一组 session 的 upstream 历史
- 编译成 block 视图
- 输出完整视图文件，供后续搜索和人工阅读使用

建议形式：

```bash
opencode-history-grep compile
```

参数说明：

- `--repository`：编译产物目录，默认 `/root/.local/share/opencode-history-grep`
- `--db`：可选覆盖 upstream SQLite 路径；默认 `/root/.local/share/opencode/opencode.db`
- `--directory`：按 session 工作目录前缀筛选
- `--since` / `--until`：按 session 有效更新时间筛选

当前默认 DB 已固定为：`/root/.local/share/opencode/opencode.db`。

### `grep`

作用：

- 在已编译视图上执行搜索
- 返回命中 block 的概要结果

建议形式：

```bash
opencode-history-grep grep --query "keyword"
```

参数说明：

- `--query`：查询单元，可重复传入多次
- `--repository`：编译产物目录，默认 `/root/.local/share/opencode-history-grep`
- `--directory`：按 session 工作目录前缀筛选
- `--since` / `--until`：按 session 有效更新时间筛选
- `--regex`：把查询解释为正则表达式，并启用跨行匹配
- `--logic`：多个 `--query` 之间的逻辑关系，目标支持 `and` / `or`，默认 `or`
- `--type`：按 block 类型筛选，可重复传入多次
- `--page`：页码，默认 1
- `--page-size`：每页条数，默认 10

类型筛选的任务化理解：

- 找需求原话 → `--type user`
- 找解释/结论 → `--type assistant`
- 找命令参数 → `--type tool_call`
- 找工具输出/错误 → `--type tool`

默认推荐优先使用 `user` / `assistant`；tool 相关筛选用于明确查命令参数或工具输出时。

当前 `grep` 的默认行为不是只查现有 repository，而是：

1. 先用默认 DB 自动 refresh compiled repository
2. 再在 refresh 后的 repository 上执行搜索

如果 session 没变化，refresh 应走增量跳过，而不是全量重编译。

单个 `--query` 默认是大小写不敏感的普通文本匹配；显式加 `--regex` 后，该 query 单元改为正则匹配，并支持跨行长内容搜索。

多个 `--query` 的目标语义：

- `--logic or`：按命中 query 数多的结果优先，但不要求全部命中
- `--logic and`：只保留同一个 session 中能命中全部 query 的结果

默认结果分页，每页 10 条，并显示页码 `a/b`。结果列表以可信 `match` 为主，不把粗糙 `summary` 当主展示字段。

如果结果过宽，CLI 应直接给出收窄提示，例如：

- 试试 `--type assistant`
- 试试 `--since/--until`
- 试试更精确短语或 `--regex`

结果排序应优先考虑：

- 多关键词命中
- 更近时间

### `show`

作用：

- 根据 block 或 anchor 回看完整上下文

建议形式：

```bash
opencode-history-grep show --session <session-id> --anchor <anchor>
```

参数说明：

- `--repository`：编译产物目录，默认 `/root/.local/share/opencode-history-grep`
- `--session`：session id
- `--anchor`：block anchor
- `--block`：按 block id 或零基 block 序号读取一条或多条 block 的完整原始内容；支持逗号分隔和重复传参。该模式先用 compiled view 定位 `message_id` / `part_id`，再回源 SQLite 读取未被 compile 摘要截断的原始 `part.data`
- `--before`：anchor 前显示多少个 block
- `--after`：anchor 后显示多少个 block
- `--all`：显示整会话 compiled view
- `--type`：配合 `--all` 使用，筛选整会话视图 block 类型；默认 `message`，即只显示 `user_text` / `assistant_text`；显式 `--type all` 才显示工具调用和工具结果
- `--from`：配合 `--all` 使用，按 block id 或零基 block 序号指定起点；省略时从开头开始
- `--to`：配合 `--all` 使用，按 block id 或零基 block 序号指定终点；省略时到结尾结束
- `--full-text`：不要截断长文本或长工具输出

这个命令不是第一优先级，但它能保证搜索结果可回跳，所以应当保留在设计里。

当前 `show` 的默认行为不是只读现有 repository，而是：

1. 先用默认 DB 自动 refresh compiled repository
2. 再从 refresh 后的 repository 回看指定 session / anchor

如果 session 没变化，refresh 应走增量跳过，而不是全量重编译。

审查类工作流应优先先读 message-only 视图，再读 all/full-text 证据视图：

```bash
opencode-history-grep show --session <session-id> --all --type message
opencode-history-grep show --session <session-id> --all --type all --full-text
opencode-history-grep show --session <session-id> --all --type all --from <block-id-or-index> --full-text
opencode-history-grep show --session <session-id> --block <block-id>,<block-id-or-index>
```

## 搜索输出要求

`grep` 的默认输出不应当是裸命中文本，而应至少包含：

- 命中 block 类型
- 命中片段
- 上下文定位信息

建议输出形态示意：

```text
page 1/3 · total 23

[assistant_text] session=ses_xxx anchor=b42
match: "history is stored in SQLite rather than Claude JSONL"
```

## 第一版输入来源优先级

状态：**已实现：第一优先级是 upstream SQLite 直读**

CLI 第一版建议支持以下两类输入，但只需先实现其中一种：

1. 直接读取 upstream SQLite 历史
2. 通过 upstream session/message API 拉取历史

当前优先级已经固定：

- 第一优先级：upstream SQLite 直读
- API 拉取只是后续补充输入层

同时还有一条固定边界：

- `system` 不进入 compiled repository，也不属于 grep 搜索域

要求已经明确：

**不能把 Claude JSONL 兼容性当成第一版前提。**

## 结果文件策略

第一版建议把编译产物写到工具自己的本地输出目录，而不是回写 upstream 数据库。

原因：

- 编译视图属于派生产物
- 搜索系统应与 upstream 原始历史解耦
- 便于后续清理、重建与调试

## 第一版成功标准

CLI 层面，当前成功标准至少包括：

1. `compile` 能稳定产出 block 视图
2. `grep` 默认返回高信号 `match` 结果
3. `show` 或等效定位机制能跳回完整上下文
4. 默认结果与仓库内容中均不包含 `system`
5. `grep` 支持分页、类型筛选、regex / 跨行查询
