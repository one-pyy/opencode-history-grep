# opencode-history-grep — 筛选设计

状态：**已实现主行为，query 逻辑待扩展**

## 目标

本文件定义 `opencode-history-grep` 在查询历史时的两个一等筛选面：

1. **目录筛选**
2. **时间筛选**

这两个筛选面是为了让模型在“跨全部 session 搜索”之外，还能把搜索范围快速收窄到更像用户真实记忆边界的集合，而不是在结果出来后再靠人工二次排除。

## 设计原则

### 1. 先缩小 session 集合，再做 block 搜索

目录与时间都不是 grep 结果上的后处理，而应当先作用到 session 选择，再在剩余 session 上做 block-aware 搜索。

原因：

- 这样结果更快，也更干净
- session 数量一旦很大，后过滤会浪费大量无意义搜索
- 这更符合“先选历史范围，再看命中”的心智模型

### 2. 筛选是查询域边界，不是输出装饰

目录与时间决定“哪些 session 允许参与搜索”，而不是只在输出里多打印几个字段。

### 3. skill 应优先教模型先筛，再 grep，再 show

这部分参考 VCC 的工作流价值，但命令语义必须是本项目自己的：

- 不是先满仓 grep 再人工读很多结果
- 而是先缩小 session 集合，再拿高信号结果回跳

## 目录筛选

### 语义

目录筛选的目标是只让**工作目录匹配指定路径前缀**的 session 进入 compile / grep / show 所依赖的 compiled repository。

这里的“目录”指 upstream session 关联的工作目录，不是 compiled repository 所在目录。

### 建议输入形式

```bash
python -m opencode_history_grep compile --db <db> --directory <path>
python -m opencode_history_grep grep --query "keyword" --repository <repo> --directory <path>
```

### 匹配规则

- 默认做**前缀匹配**，以支持“某项目根目录下的所有 session”
- 输入路径应先标准化为绝对路径
- 同一路径下的尾部斜杠差异不应改变结果

### 不做的事

- 不把目录筛选解释成文件内容过滤
- 不支持 glob 语法作为第一版必要能力
- 不要求一开始支持多个目录并集

## 时间筛选

### 语义

时间筛选的目标是只让**更新时间落在指定范围内**的 session 进入搜索域。

这不是对单条 message 或单个 block 的时间过滤，而是 session 级边界。

### 建议输入形式

```bash
python -m opencode_history_grep compile --db <db> --since 2026-04-01 --until 2026-04-22
python -m opencode_history_grep grep --query "keyword" --repository <repo> --since 2026-04-01
```

### 范围规则

- `--since`：包含该时间点之后或等于该时间点的 session
- `--until`：包含该时间点之前或等于该时间点的 session
- 第一版优先支持 ISO 8601 兼容输入
- 若只给一侧边界，另一侧视为开放区间

### 不做的事

- 不支持“最近 N 小时”这种相对时间表达作为第一版必要能力
- 不把时间筛选做成 grep 命中结果上的二次裁剪

## 对三个命令的作用边界

### `compile`

- 目录/时间筛选决定“哪些 session 会被编译进当前 repository”
- 这是生成一个**范围受控的 compiled repository** 的方法

### `grep`

- 若 repository 已经是全量编译仓库，grep 层仍可再加目录/时间筛选
- 这表示“在现有 compiled repository 中，再次缩小参与搜索的 session 集合”

### `show`

- `show` 的主输入仍是 `session + anchor`
- 第一版里，`show` 不需要额外再接目录/时间筛选
- 因为 anchor 已经唯一定位到一个 session 内 block

## 对 skill 的影响

给模型的 skill 应默认教它使用以下顺序：

1. 先判断是否需要目录筛选
2. 再判断是否需要时间筛选
3. 直接 `grep`
4. 需要显式备库时才单独 `compile`
5. 对命中结果用 `show`

如果用户只说“找一下以前的历史”，skill 不应默认全库裸搜，而应优先鼓励模型根据：

- 当前项目目录
- 用户描述的时间窗口
- 结果是否明显过宽

来决定是否先收窄范围。

## 成功标准

只有当以下几点同时满足，这两个筛选面才算真正成立：

1. CLI 参数清楚、稳定
2. session 选择先于搜索发生
3. skill 文本与真实 CLI 参数一致
4. 模型能用这两个筛选面把“历史召回”从宽泛搜索变成有边界的检索工作流
