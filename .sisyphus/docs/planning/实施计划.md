# opencode-history-grep — 实施计划

状态：**未实现**

## Goal

建立一个面向 opencode upstream 全部 session 历史的编译式检索系统：直接读取 upstream SQLite，把历史编译成接近 **VCC** 风格的 block 视图。这里的 **VCC** 指“先把会话历史编译成适合阅读和搜索的结构化视图，再在这些视图上搜索”的方法，不是普通 grep；系统通过 Python 核心库提供搜索、概要和上下文回跳能力，CLI 只是该系统的首个外壳。

## Scope

### 本次计划包含

- 全 session 枚举与读取
- SQLite 直读输入层
- 编译产物 schema 与编译仓库
- block 级搜索与排序
- 高质量概要输出
- `compile / grep / show` CLI 外壳

### 本次计划不包含

- 当前压缩链路整合
- 自动注入 prompt
- Claude JSONL 兼容
- Web/TUI
- 向量检索

## 当前状态总结

### 已确认

1. opencode upstream 历史使用自己的 SQLite 存储
2. 核心历史结构是 `session` / `message` / `part`
3. 已确认 VCC 应作为 baseline 纳入，而不是停留在抽象参考层
4. 当前架构方向已冻结为 **SQLite 直读 + 库优先 + CLI 外壳**
5. `system` 已明确排除在 compiled repository 与搜索仓库之外

### 当前缺失

1. SQLite reader 库
2. session 编译仓库
3. block schema 与编译规则实现
4. 搜索引擎与排序器
5. 高质量概要策略
6. 全 session 验证路径

## 边界定义

### 数据边界

- 原始数据源：upstream SQLite
- 派生产物：工具自己的 compiled repository
- 不回写 upstream 历史数据库
- `system` 不进入 compiled repository

### 接口边界

- 核心库提供稳定的读取、编译、搜索接口
- CLI 不直接操作 SQL，也不直接操作原始历史结构

### 组件边界

完整系统建议分为五层：

1. **reader**：扫描并读取 upstream SQLite 历史
2. **compiler**：把 message/part 编译成 block
3. **repository**：保存与刷新 compiled views
4. **search**：执行 block-aware 搜索、排序和摘要输出
5. **cli**：对外命令入口

## 关键 E2E 路径

最重要的验收路径是完整多 session 用户路径：

1. 用户执行全量或增量 `compile`
2. 系统生成 compiled repository
3. 用户执行 `grep --query <keyword>`
4. 结果按 session 与 block 返回高质量命中概要
5. 用户执行 `show --anchor <anchor>` 回看完整上下文

如果这个闭环不能成立，就说明“完整系统设计”还没有真正落地。

## 分层任务拆解

### Layer 1：SQLite reader

#### Task 1.1：定义 reader 接口

Acceptance：

- 能列出所有 session
- 能读取单个 session 的完整消息链
- 返回统一内存模型，而不是暴露 SQL 行结构

#### Task 1.2：实现全 session 枚举

Acceptance：

- 能扫描全部 session
- 能获取 session 更新时间等仓库刷新依据

### Layer 2：compiler

#### Task 2.1：定义 block schema

Acceptance：

- schema 覆盖 `user_text / assistant_text / tool_call / tool_result / compaction / subtask`
- schema 支持 `summary / search_text / display_text / anchors / session metadata`
- 不单独暴露 `reasoning` block
- `system` 不属于 schema 覆盖范围，也不进入 compiled repository

#### Task 2.0：固化 VCC baseline 与改造边界

Acceptance：

- 文档明确列出哪些编译行为直接参考 VCC
- 文档明确列出哪些层必须按本项目规则改造
- 后续实现不再用“像 VCC”这种模糊说法替代具体设计

#### Task 2.2：实现 full view 编译

Acceptance：

- 单个 session 能被编译成稳定 block 序列
- 工具调用与工具结果可读
- 角色边界明确
- tool block 默认保留调用参数
- tool result 默认截断输出
- upstream `system` 内容不会出现在编译产物中

#### Task 2.3：实现搜索视图衍生规则

Acceptance：

- search view 基于 compiled full view 生成
- 不是直接再次读取原始历史现拼

### Layer 3：compiled repository

#### Task 3.1：定义 compiled repository 目录与元数据

Acceptance：

- 编译产物有稳定目录结构
- 每个 session 的 compiled view 可独立刷新
- 仓库层持有刷新所需的最小元数据

#### Task 3.2：实现全量 compile

Acceptance：

- 能编译所有 session
- 编译失败不会破坏已有 compiled repository

#### Task 3.3：实现增量刷新策略

Acceptance：

- 能判断哪些 session 需要重编译
- 不要求一开始极致优化，但必须有清晰策略

### Layer 4：search

#### Task 4.1：实现 block-aware grep

Acceptance：

- 搜索结果以 block 为单位，不是行级命中
- 支持跨 session 搜索
- 默认搜索域不包含 `system`

#### Task 4.2：实现结果摘要与排序

Acceptance：

- 默认结果接近 VCC 风格的高信号输出
- 用户能快速判断“哪条最值得继续看”
- 同一 block 多重命中不会刷屏
- 命中 tool 输出时，只保留命中附近局部窗口，不直接展开整段输出

#### Task 4.3：实现上下文回跳协议

Acceptance：

- 每个命中结果都带稳定 anchor
- `show` 能根据 anchor 回到完整上下文

### Layer 5：CLI

#### Task 5.1：实现 `compile`

Acceptance：

- 支持全量 compile
- 支持后续增量刷新接口扩展

#### Task 5.2：实现 `grep`

Acceptance：

- 默认跨 session 搜索
- 默认输出高质量概要结果
- 默认结果中不存在 `system` 命中

#### Task 5.3：实现 `show`

Acceptance：

- 能稳定显示完整上下文
- 与 compiled repository 的 anchor 规则一致

## 依赖顺序

必须串行的部分：

1. reader 接口
2. block schema
3. full view compiler
4. compiled repository
5. search engine
6. CLI 接线

可并行的部分：

- 在 compiler 稳定后，repository 元数据和摘要策略可以并行细化
- 在 repository 稳定后，`grep` 与 `show` CLI 可以并行收尾

## 验证策略

### 单元验证

- reader：给定数据库，能正确读取 session/message/part
- compiler：给定固定历史，输出稳定 block
- repository：给定输入 session 集合，产物结构正确
- search：给定 compiled views，返回正确结果与排序

### 集成验证

- 全量 compile 后可以直接 grep
- grep 返回的 anchor 能被 show 消费

### E2E 验证

选择真实 upstream 历史，执行一个固定的抽样 E2E：

1. 从 upstream SQLite 中随机抽取 50 条原始记录
2. 抽样尽量覆盖 user / assistant / tool / compaction / subtask 等来源
3. 对这批真实记录执行 compile 与 grep 验证

必须验证：

1. compile 能把属于搜索域的记录稳定编译为 block
2. grep 能跨 session 返回高质量概要
3. `tool_call` 保留调用参数
4. `tool_result` 默认截断，命中时只保留局部窗口
5. `system` 不进入 compiled repository，因此也不会出现在 grep 结果里
6. `show` 能回到完整上下文

## 风险与待确认项

### 风险

1. 结果摘要质量不足，最终看起来仍像普通 grep
2. 全 session 编译规模上来后，仓库刷新成本过高
3. tool 输出截断窗口如果过大，结果会重新退化成大段日志；如果过小，又会失去判断依据
4. 随机抽样如果覆盖不到边缘来源类型，E2E 可能误判为通过

### 待确认

1. compiled repository 的目录结构
2. session 变更判断依据的最终选择
3. 默认 grep 输出的 session 分组方式
4. E2E 随机抽样如何补齐低频来源类型的覆盖策略

## 当前建议

当前最合理的执行顺序是：

1. 先把 reader / compiler / repository 三层定下来
2. 再打磨 search 结果质量
3. 最后接上 CLI 壳

因为这个项目已经不再是“做一个命令”，而是在做“一个完整的历史编译检索系统”。
