# opencode-history-grep — 版本边界与长期路线

状态：**未实现**

## 当前路线变更

本项目不再按“单 session 最小闭环”起步。

当前冻结的架构方向是：

- **输入来源**：upstream SQLite 直读优先
- **结果形态**：库优先 + CLI 外壳
- **目标能力**：从一开始就支持全 session 编译检索
- **结果质量目标**：接近 VCC 的编译式搜索体验

因此，版本划分不再是“从小工具慢慢长大”，而是“先按完整系统设计，再分阶段交付实现”。

## Stage A：系统骨架成立

状态：**未实现**

### 目标

先建立完整系统的四层核心骨架：

1. SQLite reader
2. compiler
3. compiled repository
4. search engine

CLI 在这个阶段只作为外壳入口，不主导内部结构。

### 必须包含

- 全 session 枚举与读取能力
- 编译产物 schema
- block 级编译模型
- 搜索输入输出协议
- CLI 壳命令接口

### 不要求完成

- 结果质量最终调优
- 增量重编译优化
- 大规模性能调优

## Stage B：VCC 风格结果质量达标

状态：**未实现**

### 目标

让搜索结果真正达到“可用”的信息质量，而不只是系统能跑。

### 必须包含

- 全 session grep
- block-aware 命中结果
- 高质量摘要
- 完整视图与命中视图之间的稳定锚点
- 工具调用 / 工具结果 / reasoning / compaction 等角色区分

### 验收信号

用户跨 session 搜索时，默认结果已经能快速判断“这是不是我要找的历史”，而不是只能看到碎片化命中。

## Stage C：索引与维护能力完善

状态：**未实现**

### 目标

让系统从“能用”进入“可持续维护、可长期使用”。

### 可进入范围

- 增量重编译
- 编译仓库刷新策略
- 搜索排序优化
- 结果导出格式稳定化
- API 输入模式作为补充输入层

### 当前仍不纳入

- 向量检索
- prompt 注入
- 压缩链路整合
- Claude JSONL 双栈兼容

这些依然不是本项目成立的前提。

## 推荐实施顺序

1. 先按完整系统拆分核心库层
2. 再让 CLI 调用这些库层
3. 然后用真实全 session 数据打磨结果质量
4. 最后才做编译仓库刷新与性能优化

## 当前决策

当前项目正式进入 **完整多 session 编译检索系统** 立项阶段。

这意味着：

- 跨 session 不是“以后再说”
- 编译仓库不是“可选优化”
- CLI 不是系统主体，而是第一层交付界面
