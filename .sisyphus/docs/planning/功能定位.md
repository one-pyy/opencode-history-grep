# opencode-history-grep — 功能定位

状态：**已实现主能力**

## 目标

这是一个放在 `my_plugins/opencode-history-grep/` 下的 Python CLI 工具。它的职责不是直接 grep 原始历史，也不是修改当前会话提示词，而是：

1. 读取 opencode upstream 的历史记录
2. 将历史记录编译成适合阅读与搜索的 block 视图
3. 在编译结果上执行结构化搜索
4. 返回带概要、角色信息与上下文指针的结果

## 非目标

以下内容**不属于**第一版范围：

- 不接入当前会话压缩链路
- 不自动把搜索结果注入 prompt
- 不兼容 Claude Code 的 `.claude/projects` JSONL 作为主输入
- 不做服务端常驻进程
- 不做 Web UI 或 TUI

## 为什么不是普通 grep

opencode upstream 的历史不是一串适合直接检索的文本，而是结构化的 session/message/part 数据。直接 grep 原始记录会出现三个问题：

1. 命中粒度太碎，难以快速理解
2. 很难一眼区分 user / assistant / tool / compaction
3. 搜索结果缺少可用概要，必须回到原始历史重新阅读

因此第一版采用“**先编译，再检索**”而不是“原始历史直接 grep”。

## 系统能力边界

当前项目不再按“单 session 第一版工具”定义，而是按完整系统目标定义：

1. 读取 upstream 全部 session 历史
2. 编译为 block 视图
3. 在编译结果上执行跨 session 搜索
4. 返回接近 VCC 风格的高质量结果

## 输入来源

状态：**已实现**

当前优先输入来源已经冻结为：

- upstream SQLite 直读

API 读取可以作为后续补充输入层，但不是当前主路径。

## 搜索域边界

状态：**已实现**

本项目的搜索仓库只接收会进入实际历史 grep 的内容。

明确规则：

- `system` **不编译进搜索仓库**
- 这不是“默认隐藏”，而是从搜索仓库边界上直接排除

原因：

- `system` 主要是运行时框架噪音，不是用户要找的历史主体
- 把它放进搜索仓库会污染命中质量、摘要质量和排序信号
- 本项目的目标是高信号历史检索，不是运行时内部事件审计

## 输出目标

当前输出目标需要覆盖两类结果：

### 1. Compiled Full View

用于完整阅读某个 session 的历史块。

### 2. Compiled Search View

用于返回命中块，并为每个命中块附带：

- block 类型
- 命中片段
- 在完整视图中的定位信息

当前结果列表默认主看 `match`，不把脆弱的一行概要当主展示字段。

## 成功标准

只有当以下三点同时满足，这个系统的第一阶段才算成立：

1. 用户可以跨 session 搜索，并快速判断“这是不是我要找的历史”
2. 用户可以从结果稳定跳到完整上下文
3. 默认结果已经是高信号摘要，而不是原始记录倾倒
