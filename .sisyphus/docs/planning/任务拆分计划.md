# opencode-history-grep — 任务拆分计划

状态：**未实现**

## 目标

把 `opencode-history-grep` 拆成可以自顶向下推进的责任层任务，确保后续实现不会把 reader、compiler、repository、search、CLI 和测试混成一团开发。

## 范围

### 本计划包含

- reader 层拆分
- compiler 层拆分
- repository 层拆分
- search 层拆分
- CLI 层拆分
- 测试层拆分

### 本计划不包含

- 具体代码实现细节
- 性能调优细节
- Web/TUI 路线
- Claude JSONL 兼容

## 当前冻结前提

在任务拆分开始前，以下边界已经确定，不再反复讨论：

1. 输入优先为 upstream SQLite
2. 项目形态是**库优先 + CLI 外壳**
3. VCC 是 baseline，但不是直接运行时宿主
4. 不做 reasoning block
5. `tool_call` 默认保留参数
6. `tool_result` 默认截断，命中时保留局部窗口
7. `system` 完全不编译进搜索仓库

## 总体执行顺序

必须按下面的顺序推进：

1. 定义 reader 输出模型
2. 定义 compiler block schema 与过滤边界
3. 实现 full view compiler
4. 定义并实现 compiled repository
5. 实现 grep/search 层
6. 实现 show 回跳层
7. 再包 CLI
8. 最后用真实 E2E 封口

原因：

- CLI 只是壳，不应该反向驱动底层设计
- 搜索结果质量取决于 compiler 和 repository，不应先写 grep 再补底层
- E2E 必须建立在完整链路之上，而不是局部单元先假装通过

## Layer 1：Reader

### 目标

把 upstream SQLite 的真实历史读取成统一内存模型，屏蔽 SQL 细节，为 compiler 提供稳定输入。

### 任务

#### 1.1 定义 reader 输出模型

内容：

- 定义 session / message / part 的项目内数据结构
- 明确哪些上游字段进入内存模型，哪些不进入

验收：

- compiler 层不需要直接读 SQL 行
- 输出模型足以表达 tool / compaction / subtask / system 等来源差异

#### 1.2 实现 session 枚举

内容：

- 列出全部 session
- 提供后续增量刷新的时间依据

验收：

- 可以稳定枚举全部 session
- 能拿到 session 级刷新所需元数据

#### 1.3 实现单 session 读取

内容：

- 读取单个 session 的 message/part 链
- 保证顺序稳定

验收：

- 单个 session 读取结果可直接供 compiler 消费

## Layer 2：Compiler

### 目标

把 reader 输出模型编译成 block 序列，并在这里冻结“什么能进入搜索仓库”。

### 任务

#### 2.1 定义 block schema

内容：

- 定义 `user_text / assistant_text / tool_call / tool_result / compaction / subtask`
- 定义 `summary / search_text / display_text / anchors / metadata`

验收：

- schema 足够表达 full view 与 search view
- `system` 不在 schema 进入范围内

#### 2.2 定义过滤与降噪规则

内容：

- 固定 `system` 排除规则
- 固定 reasoning 不单独成块规则
- 固定 tool block 展示规则

验收：

- 过滤边界和展示边界写成可实现规则，而不是口头描述

#### 2.3 实现 full view compiler

内容：

- 按时间顺序生成 block 序列
- 生成标题、概要、search_text、display_text、anchor

验收：

- 同一个 session 可稳定编译出一致 block 序列
- 输出结果可直接用于 grep 与 show

#### 2.4 实现 search view derivation

内容：

- 从 full view block 派生 search view
- 不重新回读原始 SQLite 现拼搜索结果

验收：

- search view 只依赖 compiled full view 生成

## Layer 3：Repository

### 目标

给编译结果一个稳定可刷新的仓库层，而不是每次 grep 时重新现场编译一切。

### 任务

#### 3.1 定义 compiled repository 结构

内容：

- 定义目录结构
- 定义 session 级产物组织方式
- 定义仓库元数据

验收：

- 单个 session 编译产物可独立刷新与检查

#### 3.2 实现全量 compile

内容：

- 把全部 session 编译入仓库
- 保证失败时不破坏已有可用产物

验收：

- 全量编译成功后，search 不需要再读 upstream SQLite

#### 3.3 实现增量刷新判断

内容：

- 比较 session 变更状态
- 只重编译必要 session

验收：

- 有明确的重编译判断依据

## Layer 4：Search

### 目标

在 compiled repository 上做 block-aware grep，并返回高信号结果。

### 任务

#### 4.1 实现 block-aware 匹配

内容：

- 在 `search_text` 上匹配
- 结果以 block 为单位返回

验收：

- 默认跨 session 搜索
- 不退化成行级 grep

#### 4.2 实现结果排序与去噪

内容：

- 控制同一 block 多重命中不刷屏
- 控制 session 级聚合方式
- 控制摘要优先于全文倾倒

验收：

- 默认结果能快速判断“哪条值得继续看”

#### 4.3 实现 tool 特殊显示规则

内容：

- `tool_call` 展示参数
- `tool_result` 截断展示
- 命中时输出局部窗口

验收：

- 工具结果不会因为一次命中就把大段输出全部吐出来

## Layer 5：Show / Navigation

### 目标

让 grep 结果可以回到完整上下文，而不是只剩搜索命中列表。

### 任务

#### 5.1 定义 anchor 协议

内容：

- 定义 session 内稳定 anchor
- 确保 grep 和 show 使用同一套定位规则

验收：

- 任一 grep 命中都能稳定被 show 打开

#### 5.2 实现完整上下文展示

内容：

- 根据 anchor 回跳 full view
- 展示命中 block 所在完整上下文

验收：

- show 输出与 grep 命中一致
- 完整上下文中仍不混入 `system`

## Layer 6：CLI

### 目标

给库能力包一层清楚的命令行入口，但不让 CLI 反向绑架底层设计。

### 任务

#### 6.1 `compile`

验收：

- 能触发全量 compile
- 后续可扩展增量刷新入口

#### 6.2 `grep`

验收：

- 默认跨 session 搜索
- 默认输出概要化 block 结果

#### 6.3 `show`

验收：

- 能基于 anchor 打开完整上下文

## Layer 7：Testing

### 目标

用测试把边界冻结住，避免实现阶段偷偷退化。

### 任务

#### 7.1 Reader/Compiler 单元测试

验收：

- 固定输入能产出固定 block
- `system` 被排除

#### 7.2 Repository/Search 集成测试

验收：

- compile 后 grep 可直接使用仓库结果
- show 能消费 grep 的 anchor

#### 7.3 真实历史 E2E

验收：

- 按 `testing/E2E-验证设计.md` 执行
- 随机 50 条 + 低频来源补样
- 完整验证 compile / grep / show 闭环

## 可并行与必须串行的部分

### 必须串行

1. reader 输出模型
2. compiler schema 与过滤边界
3. full view compiler
4. repository
5. search
6. show
7. CLI

### 可并行

在 compiler 稳定后：

- repository 元数据细化
- search 排序细化
- CLI 输出排版细化

在主链路稳定后：

- 单元测试补强
- E2E 样本准备工具

## 完成定义

这份任务拆分计划所定义的实现工作，只有在以下条件同时满足时才算完成：

1. 全 session 历史可以 compile 到本地仓库
2. grep 可以跨 session 返回高质量 block 结果
3. show 可以稳定回跳完整上下文
4. `tool_call` / `tool_result` 规则真实生效
5. `system` 没有进入 compiled repository
6. 真实历史 E2E 通过
